# RLPD Synthetic Demo Pipeline (Tommaso's V3 recipe)

Ported from `fedecomi04/squint:tom-separating-cubes` (commit `c6f9808`,
"Add RLPD synthetic demo pipeline + reference doc"). 50 demonstrations
generated by a scripted FK/IK controller in ~7 minutes on RTX PRO 6000.

## Output

```
/tmp/rlpd_50demos/
├── demos.h5                    # 102 MB, HDF5 v2 schema
├── meta.json
└── sanity_<color>_demoNNN.mp4  # 1 per color (6 total)
```

## Schema (`demos.h5`)

File attrs:
```
format_version = '2.0'
env_id         = 'SO101PlaceCube-v1'
control_mode   = 'pd_joint_pos'   ← ABSOLUTE joint targets
n_distractors  = 0
use_real_bowl  = True
rgb_h, rgb_w   = 80, 144
state_dim      = 21
action_dim     = 6
arm_delta_max  = 0.05
grip_delta_max = 0.20
num_demos      = 50
T              = ~785 control steps per trajectory
reward_v_min/max = -20.0 / +20.0
```

Per-demo group `demo_NNN`:
```
obs/rgb     (T, 80, 144, 3)  uint8
obs/state   (T, 21)          float32
actions     (T, 6)           float32   ABSOLUTE rad
rewards     (T,)             float32
terminals   (T,)             bool
attrs       : color_idx, cube_pos, bowl_pos, seed, success, return_sum
```

State layout (21D):
```
bowl_xyz_robot_frame(3) + target_qpos(6, = action[t-1]) + goal_color_onehot(6) + noisy_qpos(6)
```

Matches the env's `FlattenRGBDObservationWrapper` alphabetical concat,
so the demos can drop directly into the RLPD replay buffer.

## Action-space conversion (absolute → delta)

The collector deviates from spec on control_mode (absolute targets vs
the env's delta-target controller). `rlpd_utils._load_h5_v2` converts at
load time:

```python
delta[0] = act_abs[0] - QPOS_START         # first step from home pose
delta[t] = act_abs[t] - act_abs[t-1]       # successive differences
delta    = clip(delta, -DELTA_BOUND, DELTA_BOUND)    # DELTA_BOUND = [0.05]*5 + [0.2]
norm     = delta / DELTA_BOUND                       # → [-1, 1]^6
```

QPOS_START is hardcoded in the loader to match the collector's home pose
(±0.05 rad noise from the env's `initial_qpos_noise_scale` is negligible
for the first delta).

## Why "pd_joint_pos" instead of the spec's "pd_joint_target_delta_pos"

> The delta-target controller has a target-accumulation bug under
> cube/finger collision. We worked around it by collecting in
> pd_joint_pos and converting at load time.

(Flagged as `collector_deviation_control_mode` in the h5 attrs.)

Two other deviations from the spec are also flagged:
- `collector_deviation_msvecenv` — ManiSkillVectorEnv wrapper kills grasp
  dynamics under scripted IK, so demos are collected WITHOUT it.
- `collector_deviation_dr` — DR drops the IK grasp rate below 5%; demos
  are collected with `domain_randomization=False`. Training re-applies
  per-step visual jitter via `ColorJitterWrapper`.

## How to use

### 1. Collect on the Brev instance

```bash
conda activate squint
cd ~/squint-rlpd
python scripts/collect_rlpd_demos.py
# ~7 min, writes /tmp/rlpd_50demos/demos.h5
```

The script needs `h5py` and `cv2` (already in the squint env) and access
to the running ManiSkill GPU sim.

### 2. (Optional) Validate the bundle

```bash
python scripts/check_rlpd_demos_v2.py /tmp/rlpd_50demos/demos.h5
# expect: "ALL CHECKS PASSED"
# note: one attr-check fails on control_mode=pd_joint_pos — that's the
# documented deviation. The loader handles it.
```

### 3. Train RLPD with the h5

Point `--offline_path` at the h5 file. `rlpd_utils._load_h5_v2` is
selected automatically when the extension is `.h5`.

**Config A — full PlaceCube task** (matches the demos exactly: bowl in
scene, pick + lift + place reward, no skip-bowl):

```bash
python train_rlpd.py \
  --env_id=SO101PlaceCube-v1 \
  --exp_name=rlpd_with_tom_demos_fullplace_3M \
  --n_distractors=0 \
  --no-skip_bowl --use_real_bowl \
  --gamma=0.9 --num_q=10 --subset_size=2 \
  --policy_lr=3e-4 --q_lr=3e-4 --alpha_lr=3e-4 \
  --policy_frequency=4 --tau=0.01 --autotune \
  --num_envs=1024 --num_updates=512 --batch_size=512 \
  --buffer_size=500000 \
  --offline_path=/tmp/rlpd_50demos/demos.h5 \
  --offline_ratio=0.5 --offline_reward_mode=recompute \
  --no-env_domain_randomization --no-env_shadows \
  --image_height=80 --image_width=144 \
  --render_height=360 --render_width=640 \
  --sim_freq=100 --control_freq=10 \
  --apply_jitter \
  --total_timesteps=3000000 --eval_freq=200000 \
  --learning_starts=5000 --freeze_encoder_after_frac=0.9 \
  --compile --cudagraphs \
  --track --wandb_project_name=maniskill-so101 --wandb_group=RLPD \
  --wandb_entity=rayane-eth
```

**Config B — grasp-only with mismatched demos**: pass
`zero_bowl_xyz=True` via the loader API to zero out the bowl_xyz slot in
the offline state so it matches the env's skip_bowl=True padding. This
keeps the rest of the demos useful (cube approach + grasp phase). Requires
manual lerobot_kwargs override OR a follow-up CLI flag.

## Limitations / caveats

- Demos are for the **full place task** (pick + lift + place + release),
  not grasp-only. The first ~330 steps of each trajectory cover the
  approach + grasp phase, the remainder is the place phase. For
  grasp-only training (`--pick_only_reward`) the late-trajectory
  transitions are mildly off-distribution but still useful for the
  encoder (cube/gripper visual features).
- `success` is the collector's binary in-bowl check (xy distance < 5 cm
  AND cube above bowl floor), not the env's `info["success"]`. Rewards
  in the h5 are the env's dense reward stream.
- The 21D state slot for `bowl_xyz_robot_frame` is the actual bowl pose;
  if you train with `--skip-bowl`, pass `zero_bowl_xyz=True` to the
  loader so the offline buffer matches the online state distribution.
